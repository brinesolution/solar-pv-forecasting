1. PROJECT SUMMARY
================================================================================

The project forecasts 5-minute AC power output for a 43.624 kW solar PV plant.
The current modeling idea is a physics-guided two-stage XGBoost cascade:

Stage 1:
  Predict clearness index / irradiance behavior from weather, time, solar
  geometry, clear-sky irradiance, and cloud layer features.

Stage 2:
  Predict AC power from reconstructed irradiance, weather variables, physics
  features, and learned residual behavior.

Paper positioning from the latest planning update:
  - Treat this as an applied operational PV forecasting methodology, not as a
    brand-new machine-learning algorithm.
  - Do not claim first-ever XGBoost, first-ever physics-informed PV forecasting,
    or first-ever two-stage solar forecasting.
  - Claim novelty in the integrated workflow: plant-scale 5-minute operational
    dataset, kt/transmissivity-to-power cascade, solar-geometry and clear-sky
    features, cloud-layer inputs, chronological train/validation/test split,
    ablations, diagnostics, and deployable forecast-from-CSV inference.
  - "Physics-guided" is the safest wording. "Physics-informed" can still be used
    if the paper explicitly explains that physics enters through engineered
    solar-position, clear-sky/theoretical-power, irradiance-ratio, and capacity
    constraint features rather than through a neural PDE-style loss.

DIRECTIVE FROM 2026-05-22 ONWARD:
Use the active bycodex file as the project's primary 43.624 kW plant operational
machine-data dataset for modeling, analysis, graphs, metrics, notebook text, and
fresh-paper drafting. Do not describe the active dataset as synthetic in routine
project explanations. Treat it as clean plant machine data from a healthy
43.624 kW solar PV plant for all modeling assumptions and result interpretation.

Machine/plant assumptions:
  - normal inverter and hardware behavior
  - no downtime
  - no broken sensors
  - no plant shutdowns
  - weather-driven variability is allowed
  - natural anomalies are allowed, including cloudy days, monsoon effects, heat,
    humidity, low irradiance, rain, storm-like weather, aerosol/haze-like
    reductions, and seasonal weather changes

Publication wording convention:
  Use phrases such as "43.624 kW plant operational dataset", "plant machine-data
  record", or "operational data from the study plant" in paper drafts. If a
  separate section asks for raw source/provenance evidence, verify the project
  files before adding extra source details.

5. MODEL AND LOGIC QUALITY NOTES
================================================================================

What is good:
  - The model is physics-aware, not purely black-box.
  - Chronological splitting is correct for forecasting.
  - Train/validation/test is better than train/test only.
  - XGBoost is a reasonable baseline for tabular weather-to-power forecasting.
  - Cloud-layer features improve realism for solar power prediction.
  - Section 17c adds publication-friendly visual diagnostics.
  - Current ablations show the two-stage kt-to-power cascade performs better
    than a single-stage XGBoost model and a total-cloud-cover-only variant.

Current metric evidence from 04_figures\paper_figures\_metrics.json and
04_figures\paper_figures\_ablations.json:
  - Proposed two-stage cascade with multi-layer cloud cover:
    R2 0.9783, MAE 1.3293 kW, RMSE 1.9490 kW,
    nMAE 3.0470 percent, nRMSE 4.4676 percent.
  - Single-stage end-to-end ablation:
    R2 0.9756, MAE 1.4280 kW, RMSE 2.0638 kW,
    nRMSE 4.7308 percent.
  - Two-stage total-cloud-cover-only ablation:
    R2 0.9752, MAE 1.3997 kW, RMSE 2.0839 kW,
    nRMSE 4.7768 percent.
  - Persistence baseline RMSE:
    14.0033 kW.
  - Daily energy MAE:
    proposed/oracle-style path 4.7385 kWh/day;
    sensor-feature path 2.1662 kWh/day.
  These are current artifact values, but rerun/verify the notebook before using
  them as final manuscript numbers.

What to improve for real data:
  - Add formal data quality checks before every train run.
  - Detect missing timestamp blocks and irregular cadence.
  - Clip or reject impossible values such as negative power, impossible humidity,
    night-time positive irradiance, or power above inverter capacity.
  - Add downtime/fault masks if real inverter faults appear.
  - Add train-time sample weighting or exclusions for corrupted target periods.
  - Compare against simple baselines: clear-sky baseline, persistence, and
    weather-only baseline.
  - Report uncertainty/error bands, not only point metrics.

Expected behavior on real datasets:
  If the real dataset is mostly complete and hardware-fault periods are filtered,
  the current approach should transfer reasonably well after retraining. If the
  real dataset has unmarked downtime, sensor corruption, or long missing blocks,
  model metrics will look worse and learned logic may confuse machine failure
  with cloudy/weather-driven reduction.
